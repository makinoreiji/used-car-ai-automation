## 後で検討する事項

---

各機能要件の実現方式調査・検討を進めるうえで、現時点では詳細を確定しない横断論点を整理する。

ここに記載した事項は、現時点で未確定でも全体設計に進めるものとして扱う。

## 認証・認可

---

### 検討事項

- 認証基盤に Amazon Cognito を使うか
- Clerk などの認証SaaSを使うか
- 組織管理をどこまで認証基盤に寄せるか
- 組織、メンバー、ロール、権限をDB側でどう管理するか

### 現時点の扱い

プロダクト全体に必要な共通認証・認可基盤として扱う。

ただし、Cognitoを使うか、別の認証SaaSを使うかはこの時点では確定しない。

全体設計では、以下のような共通認証・認可レイヤーがある前提で仮置きする。

```text
Vercel
  ↓
認証・認可
  ↓
API Gateway
  ↓
Lambda
```

### 後で決めること

| 論点 | 決めること |
| --- | --- |
| 認証方式 | Cognitoを使うか、Clerkなどの認証SaaSを使うか |
| 組織管理 | organization / member / role / permission をどこで持つか |
| API認可 | API Gateway Authorizerでどこまで制御するか |
| Lambda内認可 | Lambda側でどこまで権限チェックするか |
| 監査 | 誰が何を実行したかをどの粒度で記録するか |

### 全体設計への影響

認証・認可方式が未確定でも、全体設計は進められる。

全体設計では、API Gateway の前段または連携先として認証・認可レイヤーを置く。

具体的なサービス選定、組織管理のDB設計、権限モデルは後続の詳細設計で決める。
