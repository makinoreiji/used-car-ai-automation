## 概要

---

画像キャプション生成機能と詳細分類の関係を整理する。

本資料は、詳細分類を導入する理由を社長レビューに出すためのメモである。

## 論点

---

画像キャプション生成には、以下の2方式がある。

| 方式 | 入力 | モデル | 特徴 |
| --- | --- | --- | --- |
| 画像を見る方式 | 画像そのもの + 前段処理で確定した車両データ | VLM | 画像を直接見てキャプション生成できるが、処理が重くコストも高くなりやすい |
| 解析結果を見る方式 | 前段処理で確定した車両データ + 画像解析結果 | LLM | 画像を再度見ないため軽く、定型的なキャプションを安定して生成しやすい |

## 方針案

---

画像キャプション生成は、後者の **解析結果を見る方式** を第一候補にする。

理由は、出品用途の画像キャプションは、画像そのものを毎回VLMに読ませなくても、前段処理で画像内容を構造化しておけば生成できる可能性が高いため。

## 詳細分類が必要になる理由

---

解析結果を見る方式を採用する場合、キャプション生成に使える粒度で前段処理の画像解析結果を残す必要がある。

例えば、以下のようなキャプションを生成したい場合、前段処理で画像の意味を分類しておく必要がある。

| 生成したいキャプション | 前段処理で必要な分類例 |
| --- | --- |
| フロント外観 | 外装 / フロント |
| リア外観 | 外装 / リア |
| サイド外観 | 外装 / サイド |
| 運転席まわり | 内装 / 運転席・ダッシュボード |
| 後部座席 | 内装 / 後部座席 |
| 荷室 | 内装 / 荷室 |
| エンジンルーム | 外装または点検系画像 / エンジンルーム |

つまり、画像キャプション生成を軽く・安定させるには、前段処理で以下を構造化しておく必要がある。

- 外装 / 内装 / 広告などの大分類
- フロント / リア / サイドなどの画角
- 運転席 / 後部座席 / 荷室 / エンジンルームなどの撮影対象
- バナー / ナンバー / 他店舗ロゴなどの加工対象有無

## 提案ロジック

---

画像キャプション生成を毎回VLMに任せると、画像そのものを都度読み込む必要があり、処理が重くなりやすい。

一方で、前段処理で画像内容を詳細分類しておけば、キャプション生成では画像そのものを見ずに、構造化された解析結果と車両データをLLMに渡せる。

このため、詳細分類は単なる画像分類精度改善ではなく、後続機能を軽く・安く・安定させるための前段構造化処理として提案できる。

## 詳細分類を導入するメリット

---

- 画像キャプション生成でVLM依存を減らせる
- LLMで定型的・安定したキャプションを生成しやすい
- 画像加工、出品データ作成、確認画面表示にも同じ解析結果を使い回せる
- 前段処理の出力を構造化することで、後続機能の実装方針を整理しやすくなる

## 現時点の結論

---

画像キャプション生成は、前段処理で確定した車両データと画像解析結果を入力にし、LLMで生成する方式を第一候補とする。

そのためには、車両画像加工機能または画像自動処理機能の前段で、キャプション生成に使える粒度の詳細分類を行う必要がある。
