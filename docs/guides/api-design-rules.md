## 概要

---

機能要件をAPI設計につなげるためのルールを整理する。

この資料は、APIの詳細仕様を決めるためのものではない。  
機能要件をどの単位で整理し、どの単位で実装・API化を検討するかを判断するためのルールブックである。

## 基本方針

---

APIは、機能名だけで切らない。

まず、利用者が何を扱いたいのかを整理する。  
そのうえで、扱う対象、対象同士の関係、対象が持つ情報、対象に対する操作を順番に整理する。

## 進め方

---

API設計につなげるために、以下の順で整理する。

1. プロダクト内のリソースを洗い出す
2. リソース同士の関係・階層を整理する
3. 各リソースのデータ構造を整理する
4. 各リソースに対する操作を洗い出す
5. リソースと操作を見て、機能の統合・分割を判断する

## 1. リソースの洗い出し

---

### 目的

プロダクトの利用者が扱いたい対象を明確にする。

### ルール

- 利用者が作成・参照・更新・削除したい対象をリソース候補にする
- 内部実装のモデル名や処理ステップは、基本的にリソースにしない
- UI部品はリソースにしない
- 同じ画像でも、利用者が扱いたい意味が違うなら別リソース候補として検討する
- ただし、最初から細かく分けすぎない

### 例

| 対象 | リソース候補にするか | 理由 |
| --- | --- | --- |
| 取込画像 | する | 利用者がアップロードし、後続処理の起点になるため |
| 車両データ | する | 利用者が確認・更新・出力したい業務データであるため |
| 画像解析結果 | する | AIの処理結果として確認・修正・再利用されるため |
| ResNet / YOLO / LaMa | しない | 利用者が直接扱う対象ではなく、内部実装のモデルであるため |
| 分類処理 / 検知処理 | しない | 処理ステップであり、利用者が管理したい業務対象ではないため |

## 2. リソース同士の関係・階層の整理

---

### 目的

リソースを親子関係にするのか、参照関係にするのかを判断する。

### 親子関係にしやすい条件

- 子リソースが親なしでは存在しにくい
- 子リソースのライフサイクルが親に従う
- 子リソースの権限が親に従う
- URL上も親の配下として扱う方が自然

### 参照関係にしやすい条件

- 将来的に別の親にも紐づく可能性がある
- 親から独立して管理したい
- ライフサイクルや権限が親と異なる
- 親子階層にすると、後から構造変更しづらくなる

### 関係自体をリソース化する条件

関係そのものに情報を持たせたい場合は、単なる参照ではなく関連リソースとして扱う。

たとえば、「この教師データセットをこの学習ジョブで使った」という関係には、以下のような情報が付く可能性がある。

- 利用日時
- 利用したデータバージョン
- 学習条件
- 評価結果
- 利用したモデル

この場合、単に教師データセットIDをモデルに持たせるのではなく、学習ジョブをリソースとして扱う方がよい。

### 例

| 関係 | 判断 |
| --- | --- |
| 車両データと取込画像 | 参照関係にしやすい。取込直後は車両との紐づきが未確定なため |
| 取込画像と画像解析結果 | 参照関係にしやすい。解析結果は複数回作られる可能性があるため |
| 車両データと出品データ | 親子または参照を検討する。出品データの独立性次第 |
| 教師データセットと学習ジョブ | 学習ジョブをリソース化する。関係自体に情報があるため |

## 3. リソースのデータ構造の整理

---

### 目的

リソースの粒度が適切か確認する。

### ルール

- リソースが持つ情報を列挙する
- 情報が多すぎる場合は、別リソースに分けることを検討する
- 情報が少なすぎる場合は、独立リソースにする必要があるか見直す
- DB設計とAPIリソース設計を混同しない
- APIリソースは利用者に見せる概念であり、DBテーブルと1対1である必要はない

### 判断観点

| 観点 | 見ること |
| --- | --- |
| 情報量 | 1つのリソースに情報が詰まりすぎていないか |
| 利用タイミング | いつ作られ、いつ参照され、いつ更新されるか |
| 更新主体 | 誰が更新するか。AIか、人間か、システムか |
| 再利用性 | 他の処理でも使う情報か |
| 履歴性 | 複数回の生成・処理結果を残す必要があるか |

### 例

画像キャプションは、取込画像に直接持たせることもできる。  
しかし、生成履歴や生成条件を残したい場合は、画像キャプションを別リソースとして扱う方がよい。

## 4. リソースに対する操作の整理

---

### 目的

各リソースに対して、利用者が何をしたいかを整理する。

### ルール

- 標準操作とカスタム操作を分ける
- 標準操作で表現できるものは、無理に独自操作にしない
- 処理の実行や生成など、標準操作に収まらないものはカスタム操作として検討する
- ただし、内部処理ステップをそのまま外部APIにしない

### 標準操作の例

| 操作 | 意味 |
| --- | --- |
| 作成 | 新しいリソースを作る |
| 取得 | 1件のリソースを見る |
| 一覧 | 複数リソースを見る |
| 更新 | 既存リソースを変更する |
| 削除 | リソースを削除する |

### カスタム操作の例

| 操作 | 意味 |
| --- | --- |
| 画像を解析する | 取込画像を起点にAI処理を開始する |
| 出品データを出力する | 出品用データをCSVや外部連携用形式で出す |
| 広告文を生成する | 車両データをもとに文章を生成する |

## 5. 機能の統合・分割の判断

---

### 目的

現在の機能カテゴリが、実装単位として自然か確認する。

### 統合を検討する条件

- 利用者から見ると1つの操作に見える
- 同じ入力と出力を扱う
- 常に同じタイミングで実行される
- 分けることで手順が増え、UXが悪くなる
- 分けても責務や権限がほとんど変わらない

### 分割を検討する条件

- 処理時間が大きく違う
- 同期処理と非同期処理が混ざっている
- 使う基盤やモデルが違う
- 権限や責務が違う
- 独立して再実行したい
- 履歴を別々に残したい
- 将来的に外部公開する単位が違う

### 今回のPJでの考え方

現時点では、APIとして外部公開する単位を確定しない。

まず機能要件をカテゴリに分ける。  
次に、カテゴリごとの実現方式を整理する。  
その結果をもとに、機能の統合・分割を判断する。

## 今回のPJで特に注意すること

---

### API化と機能分割を混同しない

機能を分けることと、外部APIとして公開することは別である。

まず内部実装の単位として自然かを考える。  
外部公開するかどうかは、その後に判断する。

### モデル名をAPI単位にしない

SageMaker EndpointやYOLOなどは、利用者が直接扱いたい対象ではない。  
モデル名ではなく、利用者がやりたい操作を起点に考える。

### 画面内操作は独立機能にしすぎない

画面上だけで完結する操作は、Vercel側のUI実装でよい。

保存、更新、実行、出力などが必要な操作は、操作対象に応じた機能へ寄せる。

### DB設計とリソース設計を混同しない

APIリソースは、利用者に見せる概念である。  
DBテーブルは、実装上の保存形式である。

APIリソースを整理した結果を、そのままDBテーブルにするとは限らない。

## 次に使う場面

---

このルールブックは、以下の作業で参照する。

- 機能要件のカテゴライズ見直し
- プロダクト実行基盤の全体設計
- API設計
- DB設計前のリソース整理
